# Skill — Draft discussion doc

Draft a short discussion document to anchor an upcoming client conversation — a working agenda the partner walks in with. A first draft to refine, not a script.

The firm's voice, identity, and hard rules are in the firm context above. Apply them.

## Input you'll get

- **Context block** — the client: company, status, what we're building (projects/phases), and recent interactions.
- **Intake** — what the conversation is for (the focus) and any notes (who's in the room, what's at stake).

## What to produce

A discussion doc in **Markdown**, ready to edit:

1. **Title + one-line purpose** — `# Discussion — <Company>` and what this conversation needs to achieve.
2. **Where things stand** — 2–4 bullets recapping the engagement and what's happened recently, from the context.
3. **Agenda** — 3–6 items, ordered, each a short phrase. The shape of the conversation.
4. **Key questions to ask** — the few questions that actually move things forward. Sharp, specific.
5. **Desired outcome** — what a good meeting produces (a decision, a next step, alignment on X).

Keep it to **one page** — skimmable right before the call. Prose + short lists.

Return only the discussion-doc Markdown — no preamble or commentary.

## Rules for this task

- **Specific and current.** Pull the real state from the context — recent interactions, project phase, open threads. A generic agenda is a failure.
- **Forward-moving.** Every item earns its place by advancing the engagement. No status-for-status's-sake.
- **Plain language.** No jargon, no banned words.

## When input is missing — never invent

Don't fabricate decisions, dates, names, or commitments. If an item needs a specific the context doesn't have, write `[NEEDS INPUT: <what's needed>]` in place and keep going. The server-side gate blocks saving anything still flagged.
