— Operations · v1.0 · 2026-05

How the ops tool works.

What each piece does, how they round-trip into the system of record, how files live alongside the database, and how to use Claude inside the tool and beside it.

— TL;DR

One source of truth, four channels in.

The ops tool is the firm's system of record. Every piece of work — a logged call, a logged hour, a sent email, a saved deliverable — writes one row to Postgres and one row to the audit log. Nothing happens silently. If it isn't tracked in the tool, it didn't happen.

Files live in Google Drive (a Shared Drive at the firm level, one folder per client). Claude shows up in three places: Quick Actions inside the ops tool, Claude Code at the per-client folder for heavy lifts, and MCP (Phase 4 — scheduled agents and CLI sessions reaching into the same DB).

— Stack

Next.js 15 (App Router) · Auth.js v5 SSO · Prisma 7 · Supabase Postgres · Vercel · Tailwind v4 · Google Drive API · Claude API (coming).

— URL

ops.shiftai.partners — Google SSO restricted to @shiftai.partners (alias @shiftcg.ai during sunset).

— 01 / System map

Four surfaces, one source of truth.

The partner browses the ops tool. Everything else — DB, Drive, Claude — sits behind it. The tool is the only place anything reads/writes through; you don't open the DB directly, you don't drop files into Drive without an Artifact row.

The relationships you'll think about most:

  • Browser ↔ Ops tool — every interaction is here. SSO restricts to firm emails.
  • Ops tool → Postgres — reads on every page render (force-dynamic), writes through server actions.
  • Ops tool → Drive — server-side via a service account that's a Content Manager on the Shared Drive. Scoped fetches and uploads only; never folder-wide scans.
  • Ops tool → Claude — Phase 3d onward. Quick Actions in the tool, then MCP for scheduled agents in Phase 4.
— 02 / Data model

Pipeline → engagement → tracking.

The main spine. A Contact belongs to a partner. A Deal is a contact in motion toward a contract. When the deal signs, it converts into a Client and a starter Project. The Client carries the engagement (revenue, Drive folder, workspace path); Projects break engagements into phases.

The fifteen models, grouped

GroupModelsWhy they matter
Identity Partner Firm members. Doubles as the "owner" reference on contacts, deals, projects.
Pipeline Contact, Deal Pre-signed relationship + sales motion.
Engagement Client, Project, Milestone, Invoice Post-signed: the actual work, billed and tracked.
Tracking Interaction, HoursEntry, Task, Artifact The four channels that round-trip every action into the DB.
Display Activity, TeamUpdate, NewsItem The "know" feed on the dashboard — non-canonical, populated for context.
Audit AuditLog Every mutation writes one row. Diligence trail; never written without an actor.
— 03 / Tracking

The four channels that round-trip back.

The principle: every channel where work happens — partner typing in the UI, a Quick Action running, a Claude Code session in a client folder, a scheduled agent — round-trips a row into the ops tool. Nothing happens silently.

ChannelModelWhat writes to it
Calls / meetings / emails Interaction Manual UI form · Quick Actions tagged AGENT · CLAUDE · Gmail/Calendar ingest (V1)
To-dos Task Manual UI form · convert-deal (creates kickoff tasks) · Quick Action suggestions
Hours HoursEntry Manual UI form · Claude Code session-end hook (V1)
Deliverables Artifact Quick Action persistence recipe · manual upload · Drive change watcher (V1)
Audit trail AuditLog Every mutation, no exceptions. Written via writeAudit().
Rule: if you do something and it doesn't write to one of the four channels above, you've done something the firm has no record of. That's the failure mode the architecture exists to prevent.
— 04 / Persistence recipe

The canonical Quick Action shape.

Every Quick Action and every Phase 5 agent follows this exact pattern. There are no exceptions. The recipe is what makes "AI surfaces" feel native to the ops tool instead of bolted on.

  1. 01
    Validate inputs. Run assertNoNeedsInput(text) on every user-facing string. If a [NEEDS INPUT — …] marker remains, the action refuses to commit. Claude never invents a price, a role, or a timeline.
  2. 02
    Resolve scope and pull context. Load the linked record (Contact / Deal / Client / Project), pick the Drive folder target, fetch any specific source files Claude needs as input.
  3. 03
    Call the model. Skill content goes in as system prompt; firm + linked-record context goes in as the user message; the result streams back. (Today: partner-templated. Phase 3d onward: real Claude API call.)
  4. 04
    Persist — in one transaction. Upload the artifact to Drive · write an Artifact row pointing at it · if it's outreach, write an Interaction row + advance lastTouchAt · write one AuditLog row via writeAudit(). Partial failures roll back.
What this buys you: partial failure can't leave the firm with a "ghost" email that was sent but not logged, or a deliverable on Drive that no Client knows about. The persistence layer is the integrity guarantee.
— 05 / Files

One Shared Drive, one folder per client.

All firm-owned files live in the Shift AI — Clients Shared Drive. A service account (ops-tool-drive@…) is a Content Manager on the Shared Drive — that single grant gives the ops tool read/write across every client folder beneath. No per-folder sharing to manage as new clients land.

Three ways the ops tool reaches files

PatternUse forHow it works
Click-out "Open Drive folder" button on a Client page — partner jumps to Drive in context. Pure UI; uses Client.driveFolderUrl. Zero server-side access.
Server-side scoped fetch Quick Actions that need a specific file ("Draft proposal" pulls last SOW). Server action calls Drive API via the service account; scoped to one or two file IDs. Never folder-wide scans.
Claude Code at the folder Heavy multi-file work — proposals, decks, build artifacts. Partner launches Claude Code at the local synced folder (Drive for Desktop). Per-client isolation: Claude can't see Beta while you're in Acme.
Per-client isolation rule: "one client at a time, not no client access." When you're in Acme — UI Quick Action or Claude Code — Claude has full read/write on Acme. The boundary prevents cross-client bleed, not Claude-to-client access.
— 06 / Claude

Three places Claude shows up.

The pattern: use the right Claude for the surface. In-tool Quick Actions for short, scoped drafts. Claude Code at the client folder for anything multi-file. MCP and scheduled agents for the recurring stuff a partner shouldn't have to remember.

— A / Quick Actions (in-tool)

Where: buttons on Contact / Client / Project pages.

For: short scoped drafts — emails, re-engagement, scope text.

How: the action loads the skill, pulls DB context, calls Claude, persists via the canonical recipe.

— B / Claude Code (at the client folder)

Where: open Claude Code in C:\…\03-Clients\<Company>\.

For: multi-file lifts — proposals, decks, build artifacts.

How: the per-client CLAUDE.md tells Claude what the engagement is. Hours hook logs time back at session end (V1).

— C / MCP server (Phase 4)

Where: any Claude Code session, any scheduled agent.

For: read/write tools — get_client, create_artifact, log_hours, update_project_status.

How: MCP server runs alongside the web app, hits the same Prisma client. Contract in docs/mcp-contract.md.

— D / Agents (Phase 5)

Where: scheduled via cron, write through MCP.

For: weekly pipeline review, stale-deal nudges, harvest-on-close.

How: agents follow the same persistence recipe as Quick Actions. No silent writes; every output is an Artifact/Interaction + AuditLog row.

The no-hallucination gate, made permanent

Two layers, both enforced. The UI marks blanks as [NEEDS INPUT — …] and disables Send. The server gate (assertNoNeedsInput()) throws if any marker reaches a persist call. The model never invents a price, a role, a date, or a commitment. If you don't supply it, it stays as an explicit marker for the partner to fill — not a guess.

— 07 / Getting started

Day one — what to set up.

For a new partner or team member. Roughly 30 min end-to-end.

  1. 01
    Workspace account. You need you@shiftai.partners to sign in. (The alias you@shiftcg.ai works during sunset.)
  2. 02
    Sign into the ops tool. Go to ops.shiftai.partners → "Sign in with Google." First sign-in auto-creates your Partner row.
  3. 03
    Drive access. An existing partner adds you to the Shift AI — Clients Shared Drive as a Content Manager. From that moment you can see every client folder.
  4. 04
    Drive for Desktop. Install Drive for Desktop, sign in with your firm account, sync the Shared Drive. The path becomes your local workspacePath for Claude Code lifts.
  5. 05
    Claude Code. Install Claude Code; sign in. To work on a client, open it at the client's local folder (cd "…/Shift AI — Clients/Acme Corp" → claude). The per-client CLAUDE.md tells it everything it needs.
  6. 06
    Walk the tool. Dashboard · Pipeline · Contacts · Clients · Projects · Invoices. Click around. Try logging hours and an interaction on a fake client. Read the audit log entries you just created.

What's load-bearing — don't drift

  • The persistence recipe. Every Quick Action and every agent uses it. No exceptions, no silent writes.
  • The audit row. writeAudit() on every mutation. One line of code; the diligence guarantee.
  • The no-hallucination gate. [NEEDS INPUT] markers, UI + server. Claude does not invent facts.
  • The Shared Drive convention. One Shared Drive, one folder per client. Don't put firm files in personal My Drives.
  • The system-of-record stance. If the work isn't tracked here, it didn't happen.
